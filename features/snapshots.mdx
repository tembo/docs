---
title: 'Snapshots'
description: 'How Tembo snapshots are created, stored, selected, and restored for sandbox VMs.'
---

Tembo uses two snapshot layers for QEMU-based sandboxes:

1. **Session snapshots**: per-VM saved state used to resume the same sandbox VM between jobs in a session flow.
2. **Golden snapshots**: reusable pre-warmed VM images used to start new VMs faster.

Container sandboxes (Docker-backed sizes) do **not** use these VM snapshots.

<Note>In the Tembo app, custom golden snapshot management is exposed on paid plans.</Note>

## Snapshot types

### Session snapshots (per VM)

Session snapshots are created from a running VM at successful job completion:

- The worker calls QEMU `savevm` (tag defaults to `latest`)
- The VM's QCOW2 overlay path and snapshot tag are stored in `VirtualMachineSnapshot` as `type: session`
- The VM is marked `Idle` and can be resumed by follow-up jobs

If saving fails, the VM is marked `Terminated` and Tembo falls back to a cold boot next time.

### Golden snapshots (reusable defaults)

Golden snapshots are built or promoted as `VirtualMachineSnapshot` rows with `type: golden`:

- Stored in cloud object storage (`VM_SNAPSHOTS_STORAGE`) and also cached on host NVMe
- Can be org-specific (`organizationId` set) or system-wide (`organizationId = null`)
- Marked active/inactive via `isActive` (active means default for its size/spec slot)

Goldens are used as the starting point for **new** VMs.

## How snapshots are restored

When a QEMU sandbox starts, Tembo attempts restore in this order:

1. Reuse a currently running VM (for keep-alive issues)
2. Restore from latest session snapshot for the VM (must be `Idle`, snapshot file must exist)
3. Start from a resolved golden snapshot (if available and compatible)
4. Cold boot from base image

Any restore failure (missing record/file, incompatible state, unavailable golden) degrades to cold boot and updates VM state safely.

## Golden snapshot selection

For a new VM, golden resolution order is:

1. **Issue-pinned snapshot** (`Issue.snapshotId`) if provided and compatible
2. Most recent compatible **org** active golden
3. Most recent compatible **system** active golden

Compatibility checks include:

- Matching VM hardware spec (`cpu`, `memoryMb`)
- Matching acceleration mode (`isKvm`: KVM vs TCG)
- Matching base image path (prevents stale-image restores)

If no compatible golden exists, Tembo cold boots.

## How goldens are created

### Build flow (manual or scheduled)

A `build_golden` worker task:

1. Creates a QCOW2 overlay from the base image
2. Boots a temporary VM
3. Optionally preloads repos/skills/dependencies
4. Saves VM state with `savevm` tag `golden`
5. Uploads QCOW2 to object storage
6. Registers the snapshot in DB
7. Marks it as the default for its size/spec slot (auto-promote)

Build options used by the product include:

- Clone selected/all connected repositories
- Embed org skills
- Pre-build dependencies from `tembo.nix`
- Target one or more QEMU sandbox sizes
- Daily rebuild schedule (cron-backed)

### Promote flow (from an existing issue VM)

Admins can promote an issue's VM snapshot to a golden:

- VM must be `Idle`
- VM is atomically marked `Terminated` before queuing promotion (prevents racing with active work)
- Worker copies the VM overlay, uploads/registers it as a golden, and caches locally
- Promoted goldens keep snapshot tag `latest` (from session save state)

## Storage model

Snapshot records live in `VirtualMachineSnapshot` with:

- `type`: `session` or `golden`
- `snapshotTag`: QEMU `savevm` tag (`latest`/`golden`)
- `spec`: `{ cpu, memoryMb, isKvm, sizeBytes }`

Session-only fields:

- `vmId`, `imagePath`, `expiresAt`

Golden-only fields:

- `storageKey`, `name`, `baseImagePath`, `isActive`

Host-local paths:

- Session overlays/snapshots: `vm.snapshotDir` (default `/var/lib/tembo/vms`)
- Golden cache: `vm.goldenDir` (default `/var/lib/tembo/golden`)

Remote storage:

- Backed by S3/Azure Blob/GCS via `vmSnapshots.storage`

## Trigger points

Snapshots are triggered by these code paths:

- **On successful sandbox jobs**: session snapshot save before job completion unblocks dependent jobs
- **On worker startup (EC2 QEMU hosts)**: system golden bootstrap for served QEMU sizes (if missing)
- **Manual "Build now"**: enqueue `build_golden`
- **Daily schedule**: scheduled queue jobs per size (`build_golden_<size>`)
- **Manual promote**: enqueue `promote_golden` for an Idle issue VM
- **Manual set-default**: switch active golden within the same size/spec slot

## Operational constraints

- VM snapshots are **QEMU-only**. Docker-backed sizes are excluded.
- Session snapshots are host-local, so follow-up jobs use **host affinity** to route back to the same worker host.
- Restore requires matching hardware/acceleration; mismatches are intentionally rejected.
- If object storage is not configured, golden snapshots are disabled and Tembo uses cold boot/session behavior.
- Golden snapshot build jobs perform memory checks and may be rejected if host memory is insufficient.

## What this improves

Golden snapshots reduce startup and setup overhead for new VMs by pre-warming repos, skills, and dependency state. Session snapshots reduce repeated setup between dependent jobs by resuming the same VM state safely.
