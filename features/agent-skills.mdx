Skills are reusable instruction bundles you manage in Tembo. Each skill gives agents a focused workflow, checklist, or set of reference files for a specific kind of work, like writing migrations, reviewing pull requests, or updating docs.

Unlike [rule files](/features/rule-files), which live in your repository and describe project wide conventions, skills are managed in Tembo settings and applied across your workspace.

## What a skill contains

Every skill has:

- A **name**
- A **description**
- A required `SKILL.md` file
- Optional extra files, such as templates, examples, or checklists

`SKILL.md` is the entry point. It should explain what the skill is for, when the agent should use it, and what steps it should follow.

The description helps Tembo and the agent understand when the skill is relevant. Keep it short and specific.

## How skills work

When you add a skill, Tembo stores it at the workspace level. On each task run, Tembo loads all workspace skills into the agent sandbox before the agent starts working.

That means:

- Skills are available on normal tasks
- Skills are available when you continue work through the [feedback loop](/features/feedback-loop)
- Skills apply across repositories in the same workspace, including multi repo tasks
- Skills are injected into the sandbox at runtime, not committed into your repository

## Create a skill

1. Open **Settings**
2. Go to **Skills**
3. Click **Add skill**
4. Enter a name
5. Add a short description that explains when the skill should be used
6. Write the instructions in `SKILL.md`
7. Add any extra files the skill needs
8. Click **Create skill**

Tembo starts each new skill with a required `SKILL.md` file. You can add more files from the same dialog.

## Writing `SKILL.md`

Write skills as direct instructions for the agent. Good skills are narrow, procedural, and easy to apply.

Example:

```markdown
# Database Migration Review

Use this skill when a task adds or changes a database migration.

## Goals

- Check that the migration matches the schema change
- Look for destructive operations
- Verify rollback or safety considerations
- Confirm application code is updated if needed

## Process

1. Read the migration files and related schema changes
2. Identify any data loss, locking, or backfill risks
3. Check whether indexes, defaults, and nullability changes are safe
4. Review code paths that depend on the changed schema
5. Summarize risks and recommend follow up changes if needed
```

## Adding extra files

Skills can include companion files alongside `SKILL.md`. These are useful for reusable assets like:

- SQL templates
- Example API payloads
- Review checklists
- Naming conventions
- Snippets the agent should copy or adapt

Example:

- `SKILL.md`
- `migration-template.sql`
- `review-checklist.md`

If the agent supports directory based skills, Tembo writes those files next to the skill entry file. If the agent does not, Tembo still provides the file contents to the agent during the run.

## Install from the marketplace

The **Skills** settings page also includes a marketplace powered by `skills.sh`.

From there you can:

- Browse popular skills
- Search for skills
- Open the original skill page on `skills.sh`
- Install a skill into your workspace

Installed marketplace skills become normal workspace skills. You can edit or delete them after install.

## How skills are provided to each agent

Tembo adapts the same skill to each supported agent's format.

### Codex

Tembo writes each skill into `.codex/skills/<skill-name>/` with `SKILL.md` as the entry file plus any companion files.

### Cursor

Tembo writes each skill into `.cursor/rules/<skill-name>/` and converts the entry file into `SKILL.mdc`. Companion files are written next to it.

### Claude Code, OpenCode, Amp, and Pi

These agents do not currently use a dedicated skill directory. Tembo appends skill content into the synthesized rule file for the run, usually `CLAUDE.md` or `AGENTS.md`, and includes any companion files inline.

If your repository already has a rule file, Tembo keeps using it and adds skill content for the task. If not, Tembo creates a temporary rule file during the run.

## Skills and rule files

Use [rule files](/features/rule-files) for durable repository context:

- Project structure
- Build and test commands
- Coding standards
- Architecture constraints

Use skills for reusable task playbooks:

- Review workflows
- Migration procedures
- Documentation update flows
- Release checklists
- Domain specific implementation guides

Most teams should use both. Put stable repo conventions in a rule file and put repeatable task specific workflows in skills.

## Editing and deleting skills

From **Settings** → **Skills**, you can:

- Edit the name, description, and file contents of any skill
- Add or remove companion files
- Delete skills you no longer want to load

When you edit a skill, Tembo replaces the full file set for that skill with the updated version.

## Best practices

- Keep each skill focused on one kind of task
- Use the description to say when the skill should be used
- Make `SKILL.md` procedural and easy to follow
- Add companion files when the agent needs exact templates or examples
- Avoid putting general project instructions in skills if they belong in a [rule file](/features/rule-files)
- Refine skills over time based on task output and [feedback loop](/features/feedback-loop) iterations
