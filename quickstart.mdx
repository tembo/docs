Use this guide to get from account setup to your first Tembo pull request.

## Before you start

- Access to the [Tembo Dashboard](https://app.tembo.io)
- A repository in GitHub, GitLab, or Bitbucket that you can authorize
- A small, well-scoped first task to run against that repository
- Optional: Slack, Linear, Jira, or [MCP servers](/integrations/mcp) if you want Tembo to use more context

<Steps>
  <Step title="Connect a repository">
    Go to [Integrations](https://app.tembo.io/integrations) and connect GitHub, GitLab, or Bitbucket. Select the repositories you want Tembo to access.
  </Step>
  <Step title="Create a task">
    Open the [Tembo Dashboard](https://app.tembo.io) and click **New Task**. Pick one or more repositories, choose your [coding agent](/features/coding-agents), and describe the work. For your first run, start with one repository and one narrow task so the result is easy to review. Tembo can also use connected integrations and [MCP servers](/integrations/mcp) for extra context during the task.
  </Step>
  <Step title="Review the PR">
    Tembo works in a [sandbox](/features/sandbox-environment), writes the code, and opens a pull request. Review it, then mention `@tembo` in PR comments to request changes.
  </Step>
  <Step title="Set up agents">
    Go to [Agents](https://app.tembo.io/automations) and pick a template or create your own with custom triggers and schedules. See [Agents](/features/agents).
  </Step>
</Steps>

## Where to tag @tembo

**Tag required:** PR comments, GitHub issues, Slack messages.

**No tag needed:** Dashboard tasks, Linear/Jira issues assigned to Tembo.

## Writing good prompts

Include: **what** needs to change, **where** (file paths), **why**, and **how you'll know it's done**.

```text
Fix memory leak in workers/job-processor.ts

Heap grows from 512MB to 4GB and crashes after ~6 hours.
Likely in cleanup logic around line 156. See Sentry #12345.

Done when: memory stable over 24+ hours, no queued jobs lost.
```

**Avoid:** "Fix the bug", "Make it faster", "Update the app and add new features". Be specific, scope to one task, and include acceptance criteria.

| Type | What to include |
|------|----------------|
| **Bug** | Steps to reproduce, expected vs actual, error messages, Sentry link |
| **Feature** | Requirements, file locations, edge cases, dependencies |
| **Refactor** | Current problems, desired state, constraints, specific files |

## Good first tasks

Start with one change that is easy to verify.

| Use case | Example prompt |
|------|----------------|
| **Basic** | Fix the typo in `docs/pricing.mdx` and update the outdated plan name in the same file. |
| **Advanced** | Add retry logic to `src/workers/email.ts`, cover the failure case with tests, and summarize any follow-up risks in the PR description. |
